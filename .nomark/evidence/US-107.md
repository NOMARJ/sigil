# Evidence: US-107 — FP adjudication endpoint wired into scan results

**Date:** 2026-06-11 · **Feature:** F-009 · **Branch:** feature/sigil-pro-fable

## Change
- **`api/routers/scan.py`**: new `POST /v1/scans/{scan_id}/findings/{finding_index}/adjudicate`. Auth via `get_current_user_unified`; gated by `require_llm_access(4)` (US-105 — Pro+ always pass, FREE metered against allowance, 402 structured denial on exhaustion). Loads the scan (404 unknown id / out-of-range index), adjudicates via `fp_adjudicator` (US-106), persists the verdict onto the finding (`finding["adjudication"]` = verdict + model + adjudicated_at) in `findings_json`, then meters usage best-effort via `credit_service.record_llm_usage(feature="fp_adjudication")` — a metering failure logs a warning, never discards a paid verdict. `LLMRefusalError` (refused on deep model AND fallback) maps to 422 `{"reason": "llm_refusal", "category"}`; refusals are not metered. Re-adjudication overwrites (idempotent).
- **`api/services/fp_adjudicator.py`**: verdict now carries `_usage` input/output token *estimates* (~4 chars/token — raw HTTP path has no usage object); endpoint pops it before persisting, so estimates never leak into stored verdicts or responses.
- **`api/main.py`** (coupled necessity, itemized): app-wide `http_exception_handler` did `str(exc.detail)`, flattening structured dict details into Python-repr strings — this silently defeated the owner-approved US-105 402 contract at the app boundary (US-105 tests exercised the dependency below the handler). Fix: dict details pass through as JSON (FastAPI default behavior); string details unchanged, including the 404 "Not Found" rewrite.
